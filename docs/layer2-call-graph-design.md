# 第2層b：コールグラフトポロジ複雑度 — 設計方針

> ステータス：CHA→RTA を実装済み。rust-analyzer/SCIP で事実抽出、tree-sitter で構築サイトを拾って RTA を精緻化（§6.2）、Tarjan で循環/ハブ検出。`konpu callgraph`（`call-graph` feature）で提供。Konpu 本体（Phase 2-A の preserve）への接続も実装済み（`konpu check --call-graph`、§7）。
> 関連：`roadmap.md` セクション1（層モデル）、セクション8（保留事項）

## 1. これは何を測るか

関数間の実際の呼び出し関係（コールグラフ）のトポロジから、循環依存・ハブへの呼び出し集中・呼び出しチェーンの深さ・密結合クラスタ等を検出する。既存の複雑度指標が測れていない領域で、循環的複雑度は関数内部の分岐、認知的複雑度はネストの深さを測るのに対し、この層は「関数同士の呼び出し関係がどれだけ絡まっているか」を測る。個々の関数が単純でも、呼び出し関係がスパゲッティならコードベース全体の理解・変更は困難になる。

Konpu本体（代数的複雑度、第3層）とは軸の性質が異なるため統合しない。tree-sitter解析基盤を共有する**兄弟ツール**として位置づけ、Konpuの層間制約機能（`konpu.toml`の`[boundaries]`、Phase 2-A）とレイヤーをまたぐ呼び出し検出の部分でのみ接点を持つ。

## 2. なぜ難しいか：動的ディスパッチと決定不能性

コールグラフの構築における本質的な難所は、呼び出し先が構文からは決定できないケースがあること。

- **trait/インターフェースの動的ディスパッチ**：`dyn Trait`へのメソッド呼び出しは、実行時までどの実装が呼ばれるか確定しない
- **ジェネリクス/高階関数**：関数ポインタやクロージャを引数に取る場合、呼び出し先は呼び出し側のコードに現れない
- **FFI越しの呼び出し、プラグインの動的ロード、関数ポインタをデータ構造に格納してから呼ぶパターン**：静的解析の対象外になりやすい

これは実装の詰めの甘さではなく、**健全かつ完全な静的コールグラフの構築は一般に決定不能**という理論的な壁がある（動的ディスパッチの解決は意味論的性質の判定であり、ライスの定理の帰結を受ける。詳細は`layer3-decidability-limits.md`の議論と同型）。したがって目標は「完全なコールグラフ」ではなく「実用上十分に安全な近似」に置く。

## 3. 採用方針：事実抽出とディスパッチ解釈の分離

構築を2段に分ける。

1. **事実抽出層**：コンパイラ/インタプリタから意味解析済みの事実（型階層、trait実装の一覧、呼び出しサイトの型情報、単相化後の呼び出し構造等）を吸い上げる
2. **解釈層**：言語のディスパッチ規則（vtable、メソッド解決順序、プロトタイプチェーン等）をモデル化し、事実に適用して「呼び出しサイトから到達しうる実装の集合」を導出する

この構成には先行事例がある。JavaのDoopはコンパイル済みバイトコードから事実をDatalogファクトとして抽出し、ポイントツー解析のルールを宣言的に記述して呼び出し先を導出する。CodeQLも同じ思想（コンパイラ統合の抽出器＋クエリ言語による解釈）。

**言語追加時の契約は「事実フォーマット」に置く。** oxidtrとKonpu本体の「アノテーションフォーマットのみが契約」という疎結合パターンと同型で、事実フォーマット（SCIPやDatalogファクトのような中間表現）さえ言語横断で定義できれば、言語追加は抽出器の追加だけで済み、ディスパッチ解釈のコアは共有できる。

## 4. 精度の選択：RTAベースの健全な過大近似

呼び出されうる対象の洗い出しには既知の精度階層がある。

| 手法 | 精度 | コスト | 健全性 |
|------|------|--------|--------|
| CHA（Class Hierarchy Analysis） | 低（そのtraitの全実装をエッジにする） | 低 | 健全（漏れなし）だが過大 |
| RTA（Rapid Type Analysis） | 中（実際にインスタンス化された型に候補を絞る） | 中 | 健全、CHAより絞られる |
| ポイントツー解析 | 高（値の流れを追跡し実到達型まで絞る） | 高（全プログラム解析でコスト爆発） | 健全だが実用コストが高い |

**採用方針：RTAを採用し、健全性の方向は「偽陽性は許容するが偽陰性は出さない」に固定する。** つまり実際には呼ばれないエッジが混ざることは許すが、実際に呼ばれるエッジを見落とすことは許さない。理由は用途がリンター/CIガードレール（循環検出・ハブ検出）であり、偽陽性による過剰報告よりも偽陰性による見落としの方が実害が大きいため。ポイントツー解析による更なる精緻化は将来の拡張候補として残すが、初期実装のスコープには含めない。

## 5. Rustにおける実装の足場

Rustは方針的にこのアプローチに適している。

- **静的ディスパッチはMIRの段階でほぼ解決済み**。ジェネリクスは単相化（monomorphization）で具体型に展開されるため、単相化後のMIRを読めばジェネリック呼び出しの解決は済んだ状態で得られる
- **動的ディスパッチ（`dyn Trait`）だけが残る**が、コンパイラはそのtraitの全実装を把握しているため、候補集合の列挙は機械的にできる（RTAの入力として使える）
- 実装経路の候補：**StableMIR**（安定インターフェースとして進行中のプロジェクト。破壊的変更のリスクは残る）、または**rust-analyzerのHIR/call hierarchy**経由

## 6. 既知の限界（消えないもの）

コンパイラ統合をもってしても消えない穴を明示しておく。

- FFI越しの呼び出し
- プラグインの動的ロード
- 関数ポインタ/クロージャをデータ構造に格納してから呼ぶパターン
- シリアライズ駆動のディスパッチ

### 6.1 SCIP 抽出における RTA 精度の天井（実装上の重要な制約）

現行の事実抽出は `rust-analyzer scip`（`konpu-cg::scip_extract`）に依存するが、rust-analyzer の SCIP 出力は occurrence の**値位置／型位置の区別・read/write role・syntax_kind をいずれも出さない**（実測: `symbol_roles=0`, `syntax_kind=UnspecifiedSyntaxKind`）。このため「型の構築（instantiation）」と「型の言及（型注釈・trait 境界・impl ヘッダ）」を区別できない。

とくに `impl Trait for T` のヘッダが必ず `T#` 参照を生むため、**trait 実装を持つ型は自身の impl ヘッダ参照だけで instantiated 集合に入る**。RTA が枝刈り対象にするのはまさに「trait 実装を持つ型」なので、結果として **RTA は実 SCIP 上で CHA に縮退する**（純粋 fixture では枝刈りを確認できるが、実コードでは縮退する）。

健全性の向き（偽陰性を出さない＝§4）を守る限り、instantiated 集合は真の構築集合の**過大近似（superset）**でなければならず、open-world（他クレート・マクロ・serde 由来の構築が見えない）では特定の型を「構築されない」と安全に断定できない。したがって **SCIP 単独では健全なまま RTA を精緻化できない**。

### 6.2 採用した精緻化: tree-sitter による構築サイト検出

SCIP だけでは縮退するため、`Facts::instantiated` を **konpu 既存の tree-sitter-rust で検出した「値位置での構築サイト」** で置き換える（`src/analyze/call_graph.rs::constructed_types`、`konpu callgraph --precision rta` 時）。捕捉するのは構造体リテラル `Foo{..}`、タプル/関連関数呼び出し `Foo(..)` / `Foo::new(..)`、および**値位置の PascalCase 識別子**（裸の unit 構造体・列挙子構築）。tree-sitter は型位置を `type_identifier` として区別するので、型注釈・`impl Trait for T` ヘッダ・ジェネリクス引数・戻り値型・trait 境界は自動的に除外され、**§6.1 の縮退要因（impl ヘッダ）が消えて RTA が実際に枝刈りする**。

トレードオフ: これは値位置の構文構築だけを見るので、**マクロ / serde / reflection 由来の構築は見えない**（§6 が既に認める穴と同種）。その範囲で偽陰性が出うる＝厳密な健全性は譲っている。完全に健全な精緻化は依然 MIR 単相化レベルの抽出（**StableMIR**）が必要で、`Facts::instantiated` はその差し替え口として分離済み（抽出器を替えれば解釈エンジン `graph` は無改変で精度が上がる）。trait 名の除外（§6.1 末尾）は SCIP 経路側の健全な掃除として残す。

将来インタプリタ言語（Python等）を扱う場合は、「コンパイラから吸い上げる」に相当するものが実行時トレースになり、これは**観測された呼び出しのみを含む過小近似**になる。静的CHA/RTAの過大近似とは意味論の方向が逆になるため、対応する場合は静的解析＋動的トレースのハイブリッドという別設計が要る。

### 6.3 Swift: tree-sitter からの直接 Facts 構築

Swift には安定した SCIP indexer を前提にできないので、SCIP を介さず **tree-sitter-swift から
`konpu_cg::Facts` を直接構築**する（`src/analyze/call_graph_swift.rs`、`call-graph` feature）。
解釈エンジン（`konpu_cg::graph`）は言語中立なので無改変で再利用する（事実抽出/解釈の分離＝§3 の
設計がそのまま効く例）。名前解決は構文ベースの過大近似（§4）: 全関数を `TraitMethod{"", bare名}`
の impl（メソッドは for_type=型、自由関数は for_type=""）、全呼び出しを `Dynamic{"", callee}`、
`Type(...)` 構築サイトを `instantiated` に。CHA=同名関数すべて、RTA=構築型（+ 常に残す自由関数）に
絞る。天井: 受け手の型が構文からは不明なのでメソッド呼びは同名全てに繋ぐ（`Money.zero()` の型限定
静的呼びの精密化は将来課題）。`konpu callgraph <dir>` は `.swift` のみのディレクトリを自動判定して
この経路を使う（Rust は従来通り SCIP）。preserve 接続（`check --call-graph`）は Swift 未対応（follow-up）。

## 7. 着手順序

Konpu本体（第3層、Phase 0〜）を優先する。この層については以下の順で進める。

1. 事実フォーマットの仕様策定（`konpu-cg::facts` として実装。Rust に必要な型・trait 実装・呼び出しサイト・インスタンス化型を持つ言語中立モデル） — **完了**
2. rust-analyzer/SCIP 経由の事実抽出器（`konpu-cg::scip_extract`。StableMIR ではなく SCIP を採用） — **完了**
3. RTAベースの解釈エンジン（`konpu-cg::graph`。CHA から開始し RTA へ拡張。循環/ハブのクエリ付き） — **完了**
4. Konpuの層間制約機能（Phase 2-A）との接続 — **完了**。`konpu check --call-graph` で
   preserve を検査。検出器 B（集約が operation に到達）+ 検出器 C（手書きマージ構築）。
   詳細は `docs/call-graph-provider-design.md` Phase 3。

現状（2026年7月）、1〜4 すべて完了。`konpu callgraph`（循環/ハブ）と
`konpu check --call-graph`（preserve）として利用可能。
