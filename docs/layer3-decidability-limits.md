# 第3層：代数的複雑度検査の決定不能性と対応方針

> ステータス：設計原理として確定。Konpuの静的検査(3.2節)・法則テスト検査(3.3節)の分割はこの制約への対応として設計されている
> 関連：`roadmap.md` セクション3.2/3.3、セクション8（保留事項）

## 1. なぜ「完全に静的検査可能」ではないのか

Konpuの4軸のうち、構造ランクと充足ギャップの核心は「法則を満たすか」の判定にある。結合律を例に取ると、これは

```
∀ a, b, c ∈ T:  (a·b)·c == a·(b·c)
```

という**全入力にわたる意味論的な等価性の主張**である。コードの構文がどうであるかではなく、コードが実行されたとき何を計算するかについての命題であり、ここに決定不能性の壁がある。

ライスの定理がそのまま適用される：「プログラムの意味論的性質（自明でないもの）を全プログラムについて判定するアルゴリズムは存在しない」。結合律の充足は明確に意味論的性質なので、任意の`op`実装に対して「これは結合的か？」に常に正しく答える静的解析器は原理的に書けない。単位律（`a·e == a`かつ`e·a == a`）、逆元律、ファンクタ則（`map(f∘g) == map(f)∘map(g)`）も全て同型の全称命題であり、同じ壁に当たる。

## 2. 具体例：構文が同一でも法則充足は分かれる

```rust
// 構文上は全く同じ形状: (T, T) -> T、純粋、閉じている
fn op1(a: i64, b: i64) -> i64 { a + b }        // 結合的
fn op2(a: i64, b: i64) -> i64 { a - b }        // 非結合的
fn op3(a: f64, b: f64) -> f64 { a + b }        // 非結合的（浮動小数点の丸め誤差）
fn op4(a: i64, b: i64) -> i64 { (a + b) % 7 }  // 結合的
fn op5(a: i64, b: i64) -> i64 {
    if a > 1000 { a } else { a + b }            // 非結合的（境界でのみ破れる）
}
```

シグネチャ・純粋性・閉包性という静的に見える情報は5つとも同一。結合律の成否は演算の**数学的内容**にのみ依存していて、それは実装の意味を完全に理解しない限り判定できない。特にop5のような「ほとんどの入力では成り立つが特定の境界で破れる」ケースは、シンボリック実行や抽象解釈を持ち込んでも一般には捕まえられない。

## 3. Konpuの対応：静的な必要条件検査 + 法則テストへの委譲

この壁への応答として、Konpuの検査は次のように分割されている（`roadmap.md` 3.2/3.3節）。

### 3.1 静的に完全判定できるもの（構文・型レベルの性質）

- 閉包性：シグネチャが`(T, T) -> T`か — 型の問題であり決定可能
- 単位元の**存在**：`Default`実装やidentity関数が定義されているか — 定義の有無は構文的事実
- 逆元操作の**存在**：同上
- 純粋性の近似：`&mut self`を取らない、等 — 構文パターンで判定可能

### 3.2 静的には必要条件しか出せないもの

- 結合律・単位律・ファンクタ則の**充足** — 「純粋で、閉じていて、浮動小数点を直接使っていない」を全部満たしても、それは結合的であることを保証しない（op2・op5が反例）。Konpuはこれをconfidenceレベルのinfoに留め、確定には法則テストを要求する

**注意すべき点：** 静的検査が保証するのは「単位元と呼ばれるものが存在する」ことまでで、「それが実際に単位元として振る舞う」ことではない。`empty()`が定義されていても`a·empty() != a`なら単位律は破れている。存在検査と法則充足の間にもギャップがある。

## 4. 法則テストも「証明」にはならない

法則テスト（proptest等のproperty-based testing）は有限個のサンプルでの反証失敗にすぎない。充足ギャップ`G = 1 - (Σpᵢ / n)`が0であることは「全法則のテストが通った」ことを意味するが、「法則が成り立つ」ことの証明ではない。op5のようなケースはランダムサンプリングが境界を踏まなければすり抜ける。

完全な保証が欲しければ定理証明器（Coq/Lean等）の領域であり、これはリンターとしてのKonpuのスコープを完全に超える。`roadmap.md`の充足ギャップの定義が「テスト充足率」として定義されているのは、この限界を暗黙に受け入れた定義になっている。

## 5. 実装上の困難（原理的な壁とは別に解決可能なもの）

「部分的に可能」と評価されるもう一つの理由は、原理的な決定不能性とは別に、実装上の困難が残るため。こちらはコンパイラ統合で解決できる。

- 型エイリアスやジェネリクスを挟んだ閉包検査（`type Pair = (T, T)`越しのシグネチャ判定）
- 別ファイル・別クレートで定義されたtrait implの発見（単位元存在検査に跨ファイル名前解決が要る）
- 文脈伝播度の計測で、フィールド型`E`の定義を辿る跨ファイル解決

これらはtree-sitter単体では対応できないが、proc macro・rust-analyzer HIR・StableMIRといったコンパイラ統合基盤（`layer2-call-graph-design.md`の事実抽出層と共有可能）で解決できる。

## 6. 「部分的に可能」の内訳まとめ

| 分類 | 内容 | 対応 |
|------|------|------|
| 決定不能（原理的に不可能） | 法則充足の静的判定（ライスの定理） | 静的検査は必要条件フィルタに留め、十分性はテストに委譲 |
| 不完全な代替 | 法則テストは反例探索であり証明ではない | 充足ギャップを「テスト充足率」として定義し、完全性を主張しない |
| 実装困難（解決可能） | 跨ファイル名前解決・型解決 | コンパイラ統合（proc macro等）で対応 |

## 7. 第2層（コールグラフ）との構造的な対比

この構図は`layer2-call-graph-design.md`で採用した「偽陽性は許すが偽陰性は出さない」方針と対になっている。

- コールグラフ（第2b層）：**過大近似**（疑わしきはエッジを引く。動的ディスパッチの解決先を漏らさない方向に倒す）
- 代数的検査（第3層）：**必要条件検査**（違反の証拠があるものだけ弾き、証明できないものはテストに送る。誤って「違反」と断定しない方向に倒す）

どちらも「決定不能な問題に対して、健全性の方向を選んで近似する」という同じ設計原理の適用になっている。前者は「見落としを許さない」方向、後者は「誤検知を許さない」方向に倒しており、これは用途の違い（コールグラフは網羅的な影響範囲把握が目的、代数的検査は確度の高い違反報告が目的）を反映した意図的な非対称性である。
